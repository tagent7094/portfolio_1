## Data Flow

### Batch Generation Pipeline (Primary Flow)
1. **Internalization** — LLM (Opus) reads founder's personality card + published posts → extracts tensions, scenes, voice markers, word count range
2. **Calibration** — LLM writes a sample paragraph in founder's voice → self-critiques for voice fidelity
3. **Anchor Inventory** — LLM (Haiku) catalogs all available "specificity anchors" from the knowledge graph
4. **Web Search Enrichment** — Retrieves trending topics for content freshness
5. **Source Selection** — LLM (Haiku) ranks viral reference posts by relevance to founder
6. **Pack Generation** (×10 sources, ×9 posts each):
   - **Dissect** — LLM analyzes the source post's structure, mechanic, and engagement hooks
   - **Transpose** — Batch A: mirror source structure with founder's argument; Batch B: divergent arguments via 13 opener mechanics
   - **Voice Validation** — 5-dimension check (vocabulary, rhythm, specificity, register, structure) with up to 2 retry passes
   - **Amplification** — Batch B openers are amplified via proven engagement mechanics
   - **Convergence Testing** — LLM detects thematically overlapping posts; targeted regen with distinct angles
   - **Quality Pass** — Dedup, anchor specificity, closer shape, mechanic distribution checks; composed regen for failures
7. **Compilation** — Final JSON + Excel output; pack history update; email notification

### Knowledge Graph Flow
1. **Ingest** — Read founder content (MD, DOCX, XLSX, PDF) → LLM-based entity extraction → Build NetworkX graph
2. **Query** — During generation, graph is queried for relevant beliefs, stories, style rules, contrast pairs
3. **Score** — Post-generation scoring against graph for influence breakdown
4. **Coverage** — Track which graph nodes are being used in generated content

## Conventions

**Naming:**
- Python: snake_case for modules, functions, variables
- React: PascalCase for components, camelCase for hooks/utilities
- Founder slugs: lowercase, underscores (e.g. `sharath`, `anish_popli`)
- API routes: `/api/{resource}` pattern

**Structure:**
- Python backend is a flat package under `src/` with domain-based subpackages
- React app mirrors the backend domain structure (batch, blog, auth, config)
- Data stored per-founder in `data/founders/{slug}/` with subdirs for graph, posts, config, run-history
- Config YAML in `config/`, with runtime overrides via admin API

**Testing:**
- Minimal — 2 test files in `tests/`, 1 in-source test (`src/graph/test_graph.py`)
- No CI test pipeline
- Quality assurance handled by the multi-stage LLM pipeline itself (voice validation, convergence, quality gate)
