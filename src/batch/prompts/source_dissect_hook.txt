# Hybrid template extraction:
#   - Batch A: the source post IS the template (extract its skeleton verbatim)

#     source's parameter slots as the parameter list for B variants

You are a structural analyst specializing in viral content mechanics. Dissect this viral post into its mechanical parts. Your output drives generation routing AND template extraction for both Batch A (mirrored) and Batch B (mechanics-only) posts.

## TASK 1 — PSYCHOLOGICAL MOVE + MECHANIC FAMILY

Figure out what the hook does to the reader — the psychological move that carries over to the generated posts. Map it to:

- `hook_mechanic_primary`: ONE of the 13 IDs in the template repository (e.g. `specific_number`, `contrarian`, `pattern_observation`, `scene_entry`, etc.). Pick the one that best matches the source's opener engine.
- `hook_sub_mechanic`: the specific sub-mechanic from the chosen family's `sub_mechanics[]` list. Examples: under `specific_number` the sub-mechanics are `first_person_monthly_cost`, `deal_size_anchor`, `age_time_anchor`, `repeated_cost_pattern`, `team_count_anchor`. Pick the most specific match.

The sub-mechanic is what determines whether the founder can mirror this post in Batch A. Sub-mechanic granularity is the difference between v6.0 (fabricated everything) and v6.1 (refuses to fabricate).

## TASK 2 — SOURCE TEMPLATE EXTRACTION (for Batch A)

Extract the source's structural skeleton. Batch A variants will mirror this skeleton with the founder's specifics. Output:

- `opener_skeleton`: the structural template of the source's first 1-3 sentences, with concrete nouns replaced by `___`. Example: `"Our ___ bill just crossed $___/month.\n\nThat's headcount territory."`
- `beat_order`: the ordered list of structural moves in the source. Examples: `["scale_anchor", "framing_recast", "diagnostic_question", "list_takedown", "stakes_enumeration", "closer"]`
- `parameter_slots`: the named slots the LLM must fill from the founder's anchors. Examples for a `first_person_monthly_cost` source: `["tool_name", "monthly_cost_usd", "team_size", "discipline_metrics"]`. Each slot must be groundable in the founder's anchor inventory.
- `narrative_arc`: the movement (e.g. `"operator_owns_spike → reveals_measurement_void"`)
- `closer_type`: the mechanic of the final paragraph (e.g. `"compounding_advantage"`, `"terminal_verdict"`, `"unresolved_tension"`, `"scene_callback"`, `"aphorism"`)

## TASK 3 — BATCH B TEMPLATE MATCH (hybrid)

For Batch B (mechanics-only — same psychological move, different vehicle), pick the closest match from the template repository AND record which of the source's parameter slots map onto that template's parameter_list. Output:

- `selected_template_id`: ID of the chosen Batch B template (one of the 13)
- `template_tier`: the tier from the repo (S/A/B/C)
- `rationale`: why this template is the closest mechanic-only fit (1 sentence)
- `narrative_engine`: from the template entry
- `closing_move`: from the template entry
- `b_parameter_mapping`: map source's parameter_slots onto this template's parameter_list where they correspond

## TASK 4 — SOURCE FITNESS CHECK (routing decision)

This determines whether the orchestrator generates 4 Batch A + 5 Batch B (default), or 9 Batch B only (when the source can't be mirrored without fabrication).

Check the founder's `anchor_inventory` for anchors that match the required sub-mechanic:

- `required_sub_mechanic`: same value as `hook_sub_mechanic` from TASK 1
- `sub_mechanic_anchor_matches`: list of `anchor_id` slugs from `anchor_inventory` whose `unlocks_sub_mechanics` field contains `required_sub_mechanic`. May be empty.
- `matching_sub_mechanic_count`: integer length of the above list
- `mirror_feasible`: boolean. True iff `matching_sub_mechanic_count ≥ 1` AND the source's mandatory parameter slot can be filled by at least one matching anchor.
- `routing_decision`: one of `"generate_4_batch_a_5_batch_b"` (default) or `"skip_batch_a_route_all_to_b"`. Use the second value ONLY when `mirror_feasible = false`. The orchestrator may override your routing (operator force-mirror) — that's fine, your decision is advisory.
- `fitness_explanation`: short string explaining the decision

## TASK 5 — ANTI-RECYCLING GUARDRAILS

The founder's existing posts are not a parts bin. Surface the patterns the transpose step must avoid copying:

- `forbidden_phrases`: list of high-signature phrases the source uses that the founder might be tempted to lift wholesale (e.g. specific number + "headcount territory" — keep the structure but force fresh phrasing in the body)
- `forbidden_templates`: list of structural patterns from the source that, if copied beat-for-beat in a Batch B post, would defeat the mechanics-only rule (e.g. `"[X] — meaningless. [Y] — inflated. [Z] — noise."` from the source's middle paragraph)

## TASK 6 — STRIP TEST CHECK

The source post itself passed the strip test (universal opener, niche body). Confirm:

- `source_strip_test_pass`: boolean
- `stripped_opener`: what remains after removing all industry-specific nouns from the source's opener
- `if_fails_reason`: empty string if pass; otherwise explain what made it niche

This signals to transpose that the founder's variants must clear the same bar.

### Special case: source unmirrorable

When `source_fitness_check.mirror_feasible = false`, set `routing_decision = "skip_batch_a_route_all_to_b"` AND populate `batch_b_template_match` with the strongest available alternative template for the founder. The orchestrator generates 9 Batch B posts instead of 4+5.

Example:

```json

  "hook_mechanic_primary": "specific_number",
  "hook_sub_mechanic": "first_person_monthly_cost",

    "selected_template_id": "pattern_observation",
```

Critical rules:
- `hook_mechanic_primary` must be one of the 13 IDs in `template_repository`.
- `hook_sub_mechanic` must appear in the chosen family's `sub_mechanics[]` list.
- `sub_mechanic_anchor_matches` must contain only `anchor_id` slugs that exist in `anchor_inventory`. Never invent anchor IDs.
- When `mirror_feasible = false`, always emit `routing_decision = "skip_batch_a_route_all_to_b"`.
- `forbidden_phrases` and `forbidden_templates` should be derived from the SOURCE POST, not from the founder's corpus (which transpose handles separately).
